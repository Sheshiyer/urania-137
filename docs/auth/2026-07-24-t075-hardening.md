## Per-finding review verdicts

### F1 — JWKS cache: kid-miss refresh correctness — **no change required**

`verifyAccessJwt` resolves the key map via `fetchJwks`; on a kid-miss it
DELETES the cached entry (never trusts a stale cache), refetches exactly once,
then fails closed with `unknown-kid` — never a third fetch, never silent
accept. Already locked by T-023 (`jwks-cache.test.ts`): hit-within-TTL → 0
refetches, rotation → exactly one refetch, kid-gone-after-refetch →
`unknown-kid`, stale-cache-not-trusted, TTL expiry → refetch, fetch
failure/HTTP error/malformed body → `jwks-unavailable`, failed fetch does not
poison the cache.

### F2 — JWKS cache: unbounded growth — **no change required (proven bounded)**

The cache is keyed by **team domain**, a value that comes only from the
`CF_ACCESS_TEAM_DOMAIN` binding — never from the request — so its cardinality
is bounded by configuration (exactly one entry in any real deployment). Each
entry's key map is **replaced wholesale** on every fetch
(`jwksCache.set(teamDomain, { keys: new Map(...) })`), never accumulated across
rotations. This was correct as written; what was missing was a test that
proves it directly. Landed: additive test aid `jwksCacheSize()` (mirrors the
existing `clearJwksCache()` aid; production never calls it) plus three
boundedness tests (see below). Failing-before: with the aid stashed, the three
boundedness tests fail (9/12 pass); passing-after: 12/12.

### F3 — Error responses leak no SELEMENE key / stack / other-user data — **no change required**

- 401/400/404 bodies are the frozen `{error, message}` envelope built from
  fixed reason discriminants (`UNAUTHORIZED`/`BAD_REQUEST`/`NOT_FOUND`) — no
  exception text, no token claims, no user data. A forged token carrying a
  victim's email is rejected without echoing it.
- `forwardToEngine`'s 502 `proxy_failed` reflects `String(err.message)` only —
  never `err.stack` — and the engine key exists solely as an upstream
  `X-API-Key` header, which cannot appear in a fetch rejection message. This
  mirrors the frozen pre-migration proxy behavior (T-033); changing it would
  be a contract change, not a hardening fix.
- Engine 4xx/5xx pass through unaltered with no key material added on the way
  back (already asserted by T-032; re-asserted here with a stack-frame regex).
- Cross-user reading access is an indistinguishable 404 (SQL-level scoping in
  `functions/lib/db.ts`); the 404 body carries zero bytes of the owner's row,
  and the row is unmutated by the attempt.

### F4 — Dev-identity guard (`DEV_IDENTITY_EMAIL` fail-closed) — **no change required**

Single fail-closed conditional (`functions/lib/dev-identity.ts`): injection
honored only when the binding var is set AND no production markers
(`CF_PAGES`, `ENVIRONMENT=production`) AND loopback hostname; client headers
play no part. Locked by T-022 (`auth-negative.test.ts`) and re-confirmed live
by the 7 bypass-negative gates below.

### F5 — Observations explicitly left alone (conservative per plan)

- `POST /api/chat/turn` streams `String(err.message)` in its error ChatEvent —
  the chat surface is owned by a parallel session (in-flight commit); out of
  scope for this pass, flagged only.
- No JWKS fetch stampede coalescing (concurrent cold-cache verifies refetch in
  parallel) — availability micro-optimization, not a correctness or security
  issue; not worth touching contract-frozen code.

## Tests added — `functions/__tests__/hardening-t075.test.ts` (12 tests)

**(A) Error-body hygiene sweep** (through `onRequest`, stubbed fetch, fake D1):
401 missing-token ×3 routes · 401 forged alg=none carrying a foreign email ·
400 malformed JSON / missing fields / bad import shape · 404 cross-user
PATCH+DELETE of a seeded foreign reading (indistinguishable, zero foreign
bytes, row unmutated) · 404 unknown route · 502 with a worst-case error
(populated `.stack`) · 504 timeout · engine 5xx passthrough. Every body
asserted: exactly `{error, message}`, correct code, no `SELEMENE_API_KEY`
substring, no stack-frame pattern (`at fn (file:line:col)`), no foreign
email/id/title/content markers.

**(B) JWKS boundedness**: 12-rotation storm → exactly 1 cache entry holding
only the latest key set (12 fetches: 1 initial + 11 kid-miss refetches) ·
10× unknown-kid attacks → size stays 1, cache never wedged (legit key verifies
after) · N distinct configured domains → exactly N entries, re-verifies add
nothing.

Failing-before/passing-after: with the `jwksCacheSize()` aid stashed out,
3 of 12 fail (the (B) block); with it restored, 12/12 pass.
